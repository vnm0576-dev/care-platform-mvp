# AGENTS.md — Care Platform

## Назначение

Этот репозиторий — канонический проект Владимира по Care Platform: Flutter-приложение и российский backend для безопасного подбора сиделок семьям пожилых людей.

Перед любой работой прочитай:

1. `docs/project-handoff.md` — актуальное состояние, решения, риски и следующий маршрут;
2. `README.md` — запуск и проверка;
3. `docs/agent-workflow.md` — GitHub-процесс;
4. `supabase/README.md` — миграции, RLS и SQL-тесты;
5. `app/README.md` — Flutter и безопасная конфигурация.

## Продуктовая стратегия

Приоритет — concierge-MVP: координатор вручную подбирает семье после выписки 2–3 проверенных сиделок, объясняет выбор и сопровождает первый выход. Публичный каталог не является главным сценарием запуска.

Границы первого MVP:

- без оплаты, чата, рейтингов и автоматического медицинского заключения;
- без загрузки медицинских документов;
- минимизация сведений о здоровье;
- ручная модерация сиделок и заявок;
- контакты не публикуются неопределённому кругу лиц без отдельного согласия.

## Техническое решение

- Flutter/Dart;
- PostgreSQL;
- Supabase Auth, PostgREST, RLS и RPC;
- целевой production: self-hosted Supabase в российском облаке;
- PostgreSQL на первом этапе — отдельная российская VM;
- файлы — private S3 в РФ через Supabase Storage API;
- Kubernetes до подтверждённой необходимости не использовать;
- основной облачный кандидат — Yandex Cloud, второй — Selectel;
- Hermes размещать на отдельной российской VM и подключать только через узкий служебный API.

Не заменяй Supabase новым backend без отдельного архитектурного решения. Не помещай `service_role`, production secrets или реальные ПДн в Flutter, GitHub, CI-логи, Telegram или зарубежные SaaS.

## Безопасность данных

Сведения о диагнозах, деменции, лекарствах, инвалидности, мобильности и потребностях ухода считать чувствительными и потенциально специальными категориями ПДн.

До юридической и ИБ-готовности:

- не добавлять выписки и медицинские документы;
- не собирать свободное клиническое описание;
- не использовать зарубежные analytics/crash/support SDK;
- не передавать ПДн в push payload;
- не делать buckets публичными;
- применять RLS на всех публикуемых таблицах;
- тестировать отрицательные сценарии доступа;
- закрывать PostgreSQL, Supabase Studio, monitoring и backups от публичного интернета.

Юридическая квалификация должна быть подтверждена российским специалистом по 152-ФЗ/323-ФЗ. Обязательны отдельные согласия, проверка полномочий родственника, уведомление Роскомнадзора, модель угроз, уровень защищённости ИСПДн и регламент инцидентов 24/72 часа.

## Процесс разработки

Всегда использовать:

`Issue → новая ветка → изменения → diff → тесты → PR → Codex review → исправления → merge`

Перед объявлением готовности:

```bash
./supabase/tests/run_local.sh
./supabase/tests/run_rls_tests.sh
./supabase/tests/run_auth_tests.sh
./supabase/tests/004_deployable_migrations_test.sh
cd app
flutter analyze
flutter test
flutter build web
```

Android AAB проверять отдельно. Наличие локальных тестов не доказывает работоспособность реального Auth или российского deployment.
