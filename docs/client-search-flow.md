# Сценарий: клиент ищет сиделку

## Цель сценария

Описать путь клиента от входа в приложение до выбора сиделки и связи с ней.

Этот сценарий нужен, чтобы дальше можно было проектировать экраны, фильтры, карточки сиделок, базу данных и логику поиска.

## Участник сценария

Роль пользователя: клиент.

Клиент ищет сиделку для родственника или подопечного, которому нужен уход дома, сопровождение, помощь после болезни, инсульта, травмы, операции или при деменции.

## Основной путь пользователя

### 1. Вход в приложение

Клиент открывает приложение.

На первом экране он видит варианты:

- войти;
- зарегистрироваться;
- выбрать роль.

Если пользователь новый, он проходит регистрацию.

### 2. Выбор роли

После регистрации клиент выбирает роль:

- Я ищу сиделку.

После выбора роли приложение переводит клиента к поиску сиделки.

### 3. Указание города

Клиент выбирает или вводит город.

Пример:

- Челябинск;
- Екатеринбург;
- Москва;
- Санкт-Петербург.

Город нужен, чтобы показывать только актуальные анкеты сиделок.

### 4. Параметры поиска

Клиент может уточнить параметры поиска.

Основные параметры:

- город;
- район;
- график;
- опыт работы;
- навыки ухода;
- опыт с деменцией;
- опыт с лежачими пациентами;
- готовность к проживанию;
- готовность к ночным сменам;
- желаемая оплата.

### 5. Список сиделок

После выбора города и параметров клиент видит список сиделок.

В списке отображаются краткие карточки.

## Карточка сиделки

Карточка сиделки должна показывать:

- ФИО;
- город;
- район;
- опыт работы;
- график;
- ключевые навыки;
- краткое описание;
- статус анкеты;
- кнопку открытия профиля.

## Подробный профиль сиделки

При открытии карточки клиент видит подробный профиль.

Профиль может включать:

- ФИО;
- фото;
- город;
- район работы;
- телефон;
- опыт работы;
- образование;
- сертификаты;
- навыки ухода;
- опыт с деменцией;
- опыт с лежачими пациентами;
- опыт после инсульта;
- опыт после инфаркта;
- опыт после травм и аварий;
- готовность к проживанию;
- готовность к ночным сменам;
- желаемую оплату;
- описание о себе.

## Способы связи

На первом этапе MVP связь может быть простой.

Возможные варианты:

- звонок по телефону;
- переход в WhatsApp;
- переход в Telegram;
- копирование номера телефона.

Встроенный чат в первой версии не обязателен.

## Завершение сценария

Сценарий считается завершённым, когда клиент:

- нашёл подходящую сиделку;
- открыл её профиль;
- выбрал способ связи;
- связался с сиделкой вне приложения или через доступный канал.

## Альтернативные ситуации

### Сиделок в городе нет

Если в выбранном городе нет сиделок, приложение показывает сообщение:

В вашем городе пока нет доступных анкет. Попробуйте изменить параметры поиска или оставить заявку.

### Слишком много результатов

Если сиделок много, клиент может использовать фильтры:

- район;
- график;
- опыт;
- навыки;
- оплата;
- проживание;
- ночные смены.

### Сиделка скрыта модератором

Если анкета скрыта или отклонена, клиент её не видит.

### Клиент не знает, кого выбрать

Приложение может показывать подсказки:

- какой опыт важен при деменции;
- что спросить у сиделки при первом звонке;
- какие документы попросить;
- как безопасно договориться о первой встрече.

## Пагинация и лимиты списка

Список сиделок должен загружаться постранично, а не одним неограниченным запросом.

Минимальное правило запроса для MVP:

```text
status = approved
city = выбранный город
limit = 20 или 50
order by approved_at desc / created_at desc
pagination: page 1, page 2, ...
```

Не использовать неограниченный `select('*')` для таблицы `caregiver_profiles`, потому что Supabase/PostgREST может ограничивать выдачу строк, и клиент увидит неполный список без очевидной ошибки.

## Статусы анкет, видимые клиенту

Клиент должен видеть только анкеты со статусом:

- approved.

Клиент не должен видеть анкеты со статусами:

- draft;
- pending_review;
- rejected;
- hidden.

## Критерии готовности сценария

Сценарий считается описанным, если понятно:

- кто пользователь;
- зачем он заходит в приложение;
- какие шаги проходит;
- какие параметры поиска использует;
- что отображается в карточке сиделки;
- что отображается в подробном профиле;
- как клиент связывается с сиделкой;
- какие альтернативные ситуации возможны.
