# Технический стек MVP v1

## Цель документа

Зафиксировать технический стек MVP v1 для приложения связи сиделок, клиентов и пансионатов.

Этот документ нужен, чтобы перед началом кода снять основные технические развилки:

- на чём писать мобильное приложение;
- где хранить данные;
- как делать авторизацию;
- где хранить фотографии и файлы;
- как организовать минимальную модерацию;
- как готовить Android-приложение к публикации.

Стек MVP v1 — это не окончательное решение навсегда.  
Это рабочее решение для первой версии, чтобы быстро собрать проверяемый продукт.

---

## Краткое решение

Для MVP v1 выбираем следующий стек:

- Mobile: Flutter;
- Language: Dart;
- Backend: Supabase;
- Database: PostgreSQL;
- Auth: Supabase Auth;
- Storage: Supabase Storage;
- Admin: сначала Supabase Table Editor или простая служебная web-админка;
- Android build: AAB для Google Play, APK/AAB для тестирования;
- Repository: GitHub;
- Workflow: Issue → Branch → Commit → Pull Request → Merge.

---

## Почему сначала фиксируем стек

До начала кода нужно определить технологическую основу проекта.

Если этого не сделать, появятся проблемы:

- сценарии будут описаны под одну архитектуру, а код начнёт делаться под другую;
- агентам будет непонятно, какие файлы создавать;
- база данных может быть спроектирована под SQL, а backend внезапно уйдёт в NoSQL;
- мобильный интерфейс может проектироваться под Flutter, а потом появится React Native;
- админка может начать расти раньше, чем появится пользовательское приложение.

Фиксация стека нужна, чтобы следующие задачи были последовательными.

---

# Выбранный стек MVP v1

## 1. Мобильное приложение

### Выбор

Flutter.

### Язык

Dart.

### Почему Flutter

Flutter подходит для MVP, потому что позволяет делать мобильное приложение с единой кодовой базой.

На первом этапе главный фокус — Android.

Позже при необходимости можно рассматривать iOS.

### Что будет в приложении

В MVP мобильное приложение должно закрывать два пользовательских сценария:

- сиделка создаёт анкету;
- клиент ищет сиделку и связывается с ней.

Администраторская часть на первом этапе не обязана быть внутри мобильного приложения.

---

## 2. Backend

### Выбор

Supabase.

### Почему Supabase

Supabase закрывает сразу несколько задач:

- хранение данных;
- авторизация пользователей;
- роли пользователей;
- хранение файлов;
- работу с PostgreSQL;
- быстрый старт без отдельного собственного сервера.

Для MVP это снижает количество технической работы.

Нам не нужно сразу писать отдельный backend с нуля.

---

## 3. База данных

### Выбор

PostgreSQL через Supabase.

### Почему PostgreSQL

Проект хорошо ложится на реляционную модель данных.

У нас уже описаны таблицы:

- users;
- caregiver_profiles;
- client_requests;
- profile_statuses;
- moderation_logs.

Между ними есть понятные связи:

- пользователь → анкета сиделки;
- пользователь → клиентская заявка;
- анкета сиделки → история модерации;
- администратор → действия модерации.

Для таких данных SQL-база удобнее, чем документная NoSQL-база.

---

## 4. Авторизация

### Выбор

Supabase Auth.

### Что нужно в MVP

В MVP нужны роли:

- caregiver — сиделка;
- client — клиент;
- admin — администратор.

Пользователь должен иметь возможность:

- зарегистрироваться;
- войти;
- выбрать роль;
- работать только со своим сценарием.

### Примечание

На первом этапе можно сделать одну основную роль на пользователя.

Позже можно добавить несколько ролей у одного пользователя.

---

## 5. Хранение файлов

### Выбор

Supabase Storage.

### Что нужно хранить

На первом этапе потенциально нужны:

- фото сиделки;
- сертификаты;
- документы, если они будут добавлены позже.

В MVP v1 загрузка документов не является обязательной.

Фото сиделки можно оставить необязательным полем.

---

## 6. Админка

### Выбор для MVP v1

Минимальная админка.

На первом этапе возможны два варианта:

1. Supabase Table Editor.
2. Простая служебная web-страница администратора.

### Почему не делаем сложную админку сразу

Сложная админка увеличит объём работы.

Для MVP важнее проверить основной механизм:

сиделка создаёт анкету → администратор проверяет → клиент видит одобренные анкеты.

Если этот механизм работает, отдельную удобную админ-панель можно развивать позже.

---

## 7. Android-публикация

### Форматы

Для Google Play нужен Android App Bundle:

- AAB.

Для локального тестирования можно использовать:

- APK;
- AAB.

### Что потребуется позже

Для публикации понадобятся:

- Google Play Developer Account;
- подпись приложения;
- название приложения;
- описание;
- иконка;
- скриншоты;
- политика конфиденциальности;
- проверка требований Google Play.

Публикация в Google Play не входит в самый первый технический шаг, но стек должен учитывать этот путь.

# Сравнение вариантов

## Flutter vs React Native

### Flutter

Плюсы:

- единый UI-подход;
- хорош для быстрого прототипирования интерфейса;
- подходит для Android и iOS;
- много готовых компонентов;
- удобно описывать экраны и переходы.

Минусы:

- нужно изучать Dart;
- часть нативных возможностей требует дополнительных пакетов;
- при сложной интеграции с платформой иногда нужны Android/iOS-навыки.

### React Native

Плюсы:

- JavaScript/TypeScript;
- большой рынок разработчиков;
- хорошая экосистема;
- можно использовать опыт web-разработки.

Минусы:

- больше зависимости от мостов и нативных модулей;
- иногда сложнее добиться стабильного поведения UI на разных устройствах;
- для текущего проекта может добавить лишнюю сложность.

### Решение

Для MVP v1 выбираем Flutter.

Причина:

Нам важнее быстро собрать понятный мобильный интерфейс и не распыляться на web-стек.

---

## Supabase vs Firebase

### Supabase

Плюсы:

- PostgreSQL;
- понятные таблицы;
- удобен для связей между сущностями;
- подходит под уже описанную схему базы;
- есть auth и storage;
- можно быстро начать без собственного backend.

Минусы:

- нужно понимать SQL и права доступа;
- потребуется аккуратно настроить политики доступа;
- часть логики может потребовать дополнительных функций позже.

### Firebase

Плюсы:

- сильная экосистема Google;
- быстрый старт;
- хорошая авторизация;
- Firestore;
- storage;
- push-уведомления;
- много документации и примеров.

Минусы:

- Firestore — документная база, а не классическая SQL;
- связи между сущностями менее очевидны;
- для нашего проекта с анкетами, статусами и модерацией SQL-модель понятнее.

### Решение

Для MVP v1 выбираем Supabase.

Причина:

У нас уже есть табличная структура данных, статусы, связи и история модерации.  
PostgreSQL лучше соответствует текущей модели проекта.

---

## Supabase vs собственный backend

### Собственный backend

Варианты:

- Node.js;
- NestJS;
- Django;
- FastAPI;
- Laravel.

Плюсы:

- полный контроль над логикой;
- гибкость;
- можно сделать сложные бизнес-правила;
- проще масштабировать нестандартную архитектуру.

Минусы:

- нужно писать API;
- нужно настраивать сервер;
- нужно делать авторизацию;
- нужно делать безопасность;
- нужно делать деплой;
- нужно вести логи;
- нужно обслуживать backend.

### Supabase

Плюсы:

- быстрее старт;
- меньше кода на первом этапе;
- сразу есть база, auth и storage;
- удобно для MVP;
- можно подключить Flutter напрямую к Supabase.

Минусы:

- часть архитектуры зависит от Supabase;
- нужно грамотно настроить права доступа;
- сложную бизнес-логику позже придётся выносить отдельно.

### Решение

Для MVP v1 выбираем Supabase без отдельного backend.

Причина:

Сейчас цель — проверить продуктовую механику, а не строить сложную серверную архитектуру.

---

## Supabase Table Editor vs отдельная админ-панель

### Supabase Table Editor

Плюсы:

- можно быстро проверять и менять данные;
- не нужно сразу писать отдельный интерфейс;
- удобно на раннем этапе;
- подходит для ручной модерации в MVP.

Минусы:

- неудобно для постоянной работы;
- риск случайно изменить лишние данные;
- не подходит для масштабной команды;
- не является полноценной админкой продукта.

### Отдельная админ-панель

Плюсы:

- удобнее для администратора;
- можно сделать кнопки "Одобрить", "Отклонить", "Скрыть";
- меньше риска ошибки;
- лучше для реальной работы.

Минусы:

- нужно тратить время на разработку;
- нужно делать авторизацию администратора;
- нужно проектировать интерфейс;
- это увеличит объём MVP.

### Решение

Для MVP v1 начинаем с минимального варианта.

Сначала можно использовать Supabase Table Editor или простую служебную страницу.

Отдельную полноценную админ-панель делаем после проверки базовой механики.

# Что входит в технический MVP v1

## Мобильное приложение

- регистрация;
- вход;
- выбор роли;
- создание анкеты сиделки;
- просмотр статуса анкеты;
- поиск сиделок клиентом;
- фильтры поиска;
- список сиделок;
- профиль сиделки;
- способы связи.

## Backend и база данных

- таблица пользователей;
- таблица анкет сиделок;
- таблица клиентских заявок;
- таблица статусов;
- таблица истории модерации;
- авторизация;
- базовые роли;
- хранение фото.

## Администрирование

- просмотр анкет на модерации;
- смена статуса анкеты;
- причина отклонения;
- причина скрытия;
- история модерации.

---

# Что не входит в MVP v1

В MVP v1 не делаем:

- встроенный чат;
- видеозвонки;
- оплату внутри приложения;
- сложные договоры;
- рейтинги и отзывы;
- автоматическую проверку документов;
- геолокацию в реальном времени;
- подбор сиделки через ИИ;
- сложную CRM;
- интеграции с Госуслугами;
- медицинскую карту пациента;
- расписание смен;
- финансовый учёт;
- полноценную панель для пансионатов.

Эти функции можно добавить позже.

---

# Архитектурный принцип MVP v1

Главная цепочка продукта:

сиделка создаёт анкету → администратор проверяет → клиент видит одобренную анкету → клиент связывается с сиделкой.

Технический стек должен обслуживать именно эту цепочку.

Всё, что не помогает этой цепочке, переносится на следующие версии.

---

# Итоговое решение

Для MVP v1 фиксируем стек:

- Flutter для мобильного приложения;
- Dart как язык мобильной разработки;
- Supabase как backend-платформа;
- PostgreSQL как база данных;
- Supabase Auth для авторизации;
- Supabase Storage для файлов;
- минимальная админка через Supabase или простую web-страницу;
- GitHub для репозитория и задач;
- Pull Request workflow для всех изменений;
- Android App Bundle для будущей публикации в Google Play.

Это решение позволяет быстрее перейти от документации к первому коду и не разрывать проект между несколькими несовместимыми вариантами.
