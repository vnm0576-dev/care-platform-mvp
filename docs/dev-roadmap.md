# Дорожная карта реализации MVP v1

## Цель документа

Описать порядок разработки MVP v1 приложения для связи сиделок, клиентов и пансионатов.

Этот документ нужен, чтобы перейти от продуктовой документации к практической разработке по понятным этапам.

Основная логика MVP:

сиделка создаёт анкету → администратор проверяет → клиент видит одобренную анкету → клиент связывается с сиделкой.

---

## Исходные решения

Для MVP v1 уже зафиксировано:

- мобильное приложение: Flutter;
- язык: Dart;
- backend: Supabase;
- база данных: PostgreSQL;
- авторизация: Supabase Auth;
- хранение файлов: Supabase Storage;
- репозиторий: GitHub;
- рабочий процесс: Issue → Branch → Commit → Pull Request → Merge.

---

# Общий план разработки

Разработка MVP v1 делится на этапы:

0. Подготовка репозитория и технической базы.
1. Подготовка Supabase-проекта.
2. Создание базы данных.
3. Настройка RLS-политик Supabase.
4. Настройка авторизации.
5. Реализация минимальной модерации.
6. Подготовка Flutter-проекта.
7. Реализация роли сиделки.
8. Реализация роли клиента.
9. Тестирование MVP.
10. Подготовка Android-сборки.
11. Подготовка к публикации.

Главная идея:

сначала фиксируем данные, роли, статусы и доступы; потом переходим к Flutter.

Она отвечает за авторизацию.

Именно profiles должна хранить роль, имя, телефон и данные, которые нужны приложению.

## Таблицы MVP

Нужно создать таблицы:

- profiles;

- moderation_logs.

## Задачи

- создать SQL-скрипт начальной схемы;
- создать таблицу profiles;
- связать profiles с auth.users;
- создать таблицу caregiver_profiles;
- создать таблицу client_requests;
- создать таблицу profile_statuses;
- создать таблицу moderation_logs;
- добавить связи между таблицами;
- добавить базовые статусы анкет;
- проверить создание тестовых данных.

## Статусы анкет

В MVP используются статусы:

- draft;
- pending_review;
- approved;
- rejected;
- hidden.

## Почему pending_review, а не pending

pending_review точнее, потому что означает ожидание именно модерации.

Это название нужно использовать одинаково:

- в документации;
- в SQL;
- во Flutter-моделях;
- в фильтрах;
- в RLS-политиках;
- в логике модерации.

# Этап 3. Настройка RLS-политик Supabase

## Цель

Настроить Row Level Security, чтобы пользователи видели и изменяли только те данные, к которым у них есть доступ.

Для Supabase это обязательная часть архитектуры, а не дополнительная настройка.

## Основные правила доступа

### Сиделка

Сиделка может:

- видеть свой профиль;
- редактировать свой профиль;
- создавать свою анкету;
- видеть свою анкету в любом статусе;
- редактировать свою анкету до отправки на модерацию;
- повторно отправлять отклонённую анкету на модерацию.

Сиделка не может:

- видеть чужие черновики;
- редактировать чужие анкеты;
- менять статус анкеты напрямую;
- писать в moderation_logs.

### Клиент

Клиент может:

- видеть только анкеты со статусом approved;
- использовать фильтры поиска;
- открывать профиль одобренной сиделки;
- создавать клиентскую заявку, если эта функция включена.

Клиент не может:

- видеть анкеты draft;
- видеть анкеты pending_review;
- видеть анкеты rejected;
- видеть анкеты hidden;
- менять статусы анкет;
- писать в moderation_logs.

### Администратор

Администратор может:

- видеть анкеты на модерации;
- видеть анкеты approved;
- видеть анкеты rejected;
- видеть анкеты hidden;
- менять статус анкеты;
- указывать причину отклонения;
- указывать причину скрытия;
- писать действия в moderation_logs.

Администратор должен работать через минимальную админ-функцию или временно через Supabase Table Editor.

## Минимальные RLS-правила

Нужно описать и реализовать политики:

- сиделка видит и редактирует только свою анкету;
- клиент видит только approved-анкеты;
- админ видит pending_review, approved, rejected, hidden;
- обычный пользователь не может менять status;
- только админ может писать moderation_logs;
- пользователь может читать и обновлять только свой profiles;
- роль пользователя хранится в profiles.

## Критерий готовности

В Supabase включены RLS-политики для основных таблиц, и обычный пользователь не может получить лишний доступ.

# Этап 5. Минимальная модерация

## Цель

Сделать проверку анкет перед публикацией.

## Важное ограничение

Supabase Table Editor можно использовать только для первичной ручной проверки механики.

Для реального MVP нужна минимальная админ-функция изменения статуса с записью действия в moderation_logs.

## Временный вариант

На самом раннем этапе администратор может вручную проверить механику через Supabase Table Editor.

Допустимые изменения статуса:

- pending_review → approved;
- pending_review → rejected;
- approved → hidden.

## Рабочий вариант для MVP

Для MVP нужно сделать минимальную админ-функцию:

- вывести список анкет со статусом pending_review;
- открыть анкету;
- одобрить анкету;
- отклонить анкету;
- указать причину отклонения;
- скрыть опубликованную анкету;
- указать причину скрытия;
- записать действие в moderation_logs.

## Критерий готовности

Анкета не появляется у клиента, пока администратор не присвоит ей статус approved.

Каждое решение администратора сохраняется в moderation_logs.

# Что не делаем до первого рабочего MVP

До первого рабочего MVP не делаем:

- встроенный чат;
- оплату;
- рейтинги;
- отзывы;
- ИИ-подбор;
- сложную CRM;
- отдельный личный кабинет пансионата;
- автоматическую проверку документов;
- геолокацию в реальном времени;
- сложную админ-панель;
- push-уведомления;
- видеозвонки;
- финансовый учёт;
- расписание смен;
- интеграции с Госуслугами.

Эти функции можно добавить после проверки базовой механики.

# Критерий готовности MVP v1

MVP v1 можно считать готовым, если:

1. Сиделка может зарегистрироваться.
2. Для сиделки создаётся запись в profiles.
3. Сиделка может создать анкету.
4. Анкета уходит на модерацию со статусом pending_review.
5. Администратор может одобрить анкету.
6. Решение администратора записывается в moderation_logs.
7. Клиент видит только approved-анкету.
8. Клиент открывает профиль сиделки.
9. Клиент видит способ связи.
10. Обычный пользователь не может изменить status анкеты.
11. Приложение запускается на Android-устройстве.

# Главный принцип реализации

Каждый следующий шаг должен быть маленькой задачей.

Одна задача — одна ветка — один pull request.

Не смешивать в одном pull request:

- документацию;
- Flutter-код;
- SQL-схему;
- RLS-политики;
- настройки Supabase;
- Android-сборку.

Так проект останется управляемым.
