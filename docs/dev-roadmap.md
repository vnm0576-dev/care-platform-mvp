# Дорожная карта реализации MVP v1

## Цель документа

Описать порядок разработки MVP v1 приложения для связи сиделок, клиентов и пансионатов.

Этот документ нужен, чтобы перейти от продуктовой документации к практической разработке без хаоса.

Основная логика MVP:

сиделка создаёт анкету → администратор проверяет → клиент видит одобренную анкету → клиент связывается с сиделкой.

---

## Исходные решения

Для MVP v1 уже зафиксировано:

- мобильное приложение: Flutter;
- язык: Dart;
- backend: Supabase;
- база данных: PostgreSQL;
- авторизация: Supabase Auth;
- хранение файлов: Supabase Storage;
- репозиторий: GitHub;
- рабочий процесс: Issue → Branch → Commit → Pull Request → Merge.

---

# Этап 7. Минимальная модерация

## Цель

Сделать проверку анкет перед публикацией.

## Первый вариант

На самом раннем этапе можно использовать Supabase Table Editor.

Администратор вручную меняет статус анкеты:

- pending → approved;
- pending → rejected;
- approved → hidden.

## Второй вариант

После проверки механики можно сделать простую служебную web-страницу или экран администратора.

## Задачи

- вывести список анкет со статусом pending;
- открыть анкету;
- одобрить анкету;
- отклонить анкету;
- указать причину отклонения;
- скрыть опубликованную анкету;
- записать действие в moderation_logs.

## Критерий готовности

Анкета не появляется у клиента, пока администратор не присвоит ей статус approved.

# Что не делаем до первого рабочего MVP

До первого рабочего MVP не делаем:

- встроенный чат;
- оплату;
- рейтинги;
- отзывы;
- ИИ-подбор;
- сложную CRM;
- отдельный личный кабинет пансионата;
- автоматическую проверку документов;
- геолокацию в реальном времени;
- сложную админ-панель;
- push-уведомления.

Эти функции можно добавить после проверки базовой механики.

---

# Критерий готовности MVP v1

MVP v1 можно считать готовым, если:

1. Сиделка может зарегистрироваться.
2. Сиделка может создать анкету.
3. Анкета уходит на модерацию.
4. Администратор может одобрить анкету.
5. Клиент видит одобренную анкету.
6. Клиент открывает профиль сиделки.
7. Клиент видит способ связи.
8. Приложение запускается на Android-устройстве.

# Главный принцип реализации

Каждый следующий шаг должен быть маленькой задачей.

Одна задача — одна ветка — один pull request.

Не смешивать в одном pull request:

- документацию;
- Flutter-код;
- SQL-схему;
- настройки Supabase;
- Android-сборку.

Так проект останется управляемым.
