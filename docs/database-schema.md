# Схема базы данных MVP v1

## Цель документа

Описать базовую структуру данных для MVP приложения связи сиделок, клиентов и пансионатов.

Схема основана на трёх ключевых сценариях:

- сиделка создаёт анкету;
- клиент ищет сиделку;
- администратор модерирует анкету.

На первом этапе цель не в том, чтобы сделать идеальную промышленную базу данных, а в том, чтобы зафиксировать понятную структуру для будущей реализации в Supabase/PostgreSQL.

---

## Основные сущности

В MVP используются следующие таблицы:

- users;
- caregiver_profiles;
- client_requests;
- profile_statuses;
- moderation_logs.

---

## users

Таблица пользователей приложения.

В этой таблице хранятся базовые данные всех пользователей: сиделок, клиентов и администраторов.

### Поля

- id — уникальный идентификатор пользователя;
- email — email пользователя;
- phone — телефон пользователя;
- role — роль пользователя;
- created_at — дата создания пользователя;
- updated_at — дата последнего обновления.

### Возможные роли

- caregiver — сиделка;
- client — клиент;
- admin — администратор.

### Примечание

Один пользователь может иметь одну основную роль в MVP.

В будущих версиях можно добавить возможность нескольких ролей у одного пользователя.

---

## caregiver_profiles

Таблица анкет сиделок.

В этой таблице хранится основная информация, которую видит клиент после одобрения анкеты администратором.

### Поля

- id — уникальный идентификатор анкеты;
- user_id — ссылка на пользователя из таблицы users;
- full_name — ФИО сиделки;
- city — город;
- district — район работы;
- contact_phone — контактный телефон;
- experience — опыт работы;
- education — образование;
- certificates — сертификаты;
- skills — навыки ухода;
- schedule — желаемый график;
- description — краткое описание о себе;
- desired_payment — желаемая оплата;
- ready_for_live_in — готовность к проживанию;
- ready_for_night_shifts — готовность к ночным сменам;
- dementia_experience — опыт с деменцией;
- bedridden_experience — опыт с лежачими пациентами;
- stroke_experience — опыт после инсульта;
- heart_attack_experience — опыт после инфаркта;
- trauma_experience — опыт после травм и аварий;
- photo_url — ссылка на фото;
- status — текущий статус анкеты;
- rejection_reason — причина отклонения;
- hidden_reason — причина скрытия;
- created_at — дата создания анкеты;
- updated_at — дата последнего обновления;
- submitted_at — дата отправки на модерацию;
- approved_at — дата одобрения;
- rejected_at — дата отклонения;
- hidden_at — дата скрытия.

### Связи

- caregiver_profiles.user_id связан с users.id.
- Один пользователь с ролью caregiver может иметь одну активную анкету.

---

## client_requests

Таблица клиентских заявок.

Эта таблица нужна, если клиент не просто смотрит список сиделок, а оставляет заявку на подбор.

На первом этапе MVP эта таблица может быть простой.

### Поля

- id — уникальный идентификатор заявки;
- user_id — ссылка на пользователя из таблицы users;
- city — город;
- district — район;
- care_type — тип ухода;
- description — описание ситуации;
- contact_phone — контактный телефон клиента;
- preferred_schedule — желаемый график;
- desired_payment — желаемая оплата;
- needs_live_in — нужен ли уход с проживанием;
- needs_night_shifts — нужны ли ночные смены;
- dementia_case — есть ли деменция;
- bedridden_case — лежачий ли пациент;
- stroke_case — состояние после инсульта;
- heart_attack_case — состояние после инфаркта;
- trauma_case — состояние после травмы или аварии;
- created_at — дата создания заявки;
- updated_at — дата последнего обновления.

### Связи

- client_requests.user_id связан с users.id.
- Один клиент может создать несколько заявок.

---

## profile_statuses

Справочник статусов анкет сиделок.

Эта таблица описывает возможные статусы анкеты.

### Поля

- code — код статуса;
- title — название статуса;
- description — описание статуса;
- visible_to_client — виден ли этот статус клиенту;
- visible_to_caregiver — виден ли этот статус сиделке;
- visible_to_admin — виден ли этот статус администратору.

### Статусы

#### draft

Черновик.

Анкета создана, но ещё не отправлена на модерацию.

Клиент её не видит.

#### pending

Ожидает модерации.

Сиделка отправила анкету на проверку администратору.

Клиент её не видит.

#### approved

Опубликована.

Администратор проверил и одобрил анкету.

Клиент видит эту анкету в списке сиделок.

#### rejected

Отклонена.

Администратор отклонил анкету и указал причину.

Клиент её не видит.

#### hidden

Скрыта администратором.

Анкета может быть скрыта после публикации из-за жалобы, ошибки, подозрения на спам или нарушения правил.

Клиент её не видит.

---

## moderation_logs

Таблица истории модерации.

Эта таблица нужна, чтобы хранить действия администратора по анкетам сиделок.

### Поля

- id — уникальный идентификатор записи;
- caregiver_profile_id — ссылка на анкету сиделки;
- admin_user_id — ссылка на администратора из таблицы users;
- old_status — предыдущий статус анкеты;
- new_status — новый статус анкеты;
- reason — причина решения;
- comment — дополнительный комментарий администратора;
- created_at — дата действия.

### Связи

- moderation_logs.caregiver_profile_id связан с caregiver_profiles.id.
- moderation_logs.admin_user_id связан с users.id.

### Примеры действий

Администратор может:

- одобрить анкету;
- отклонить анкету;
- скрыть анкету;
- вернуть анкету на доработку;
- повторно одобрить исправленную анкету.

---

## Правила видимости

## Что видит клиент

Клиент видит только анкеты со статусом:

- approved.

Клиент не видит анкеты со статусами:

- draft;
- pending;
- rejected;
- hidden.

## Что видит сиделка

Сиделка видит свою анкету в любом статусе:

- draft;
- pending;
- approved;
- rejected;
- hidden.

Также сиделка должна видеть причину отклонения, если анкета получила статус rejected.

## Что видит администратор

Администратор видит все анкеты во всех статусах:

- draft;
- pending;
- approved;
- rejected;
- hidden.

Также администратор видит историю модерации.

---

## Основные связи между таблицами

### users → caregiver_profiles

Один пользователь с ролью caregiver может иметь одну анкету сиделки.

Связь:

users.id → caregiver_profiles.user_id

### users → client_requests

Один пользователь с ролью client может иметь несколько клиентских заявок.

Связь:

users.id → client_requests.user_id

### caregiver_profiles → moderation_logs

Одна анкета сиделки может иметь много записей истории модерации.

Связь:

caregiver_profiles.id → moderation_logs.caregiver_profile_id

### users → moderation_logs

Один пользователь с ролью admin может выполнить много действий модерации.

Связь:

users.id → moderation_logs.admin_user_id

---

## Минимальные поля для публикации анкеты

Анкета сиделки не может быть опубликована, если не заполнены:

- full_name;
- city;
- contact_phone;
- experience;
- schedule;
- skills;
- description.

Если хотя бы одно обязательное поле не заполнено, анкета не должна получать статус approved.

---

## Логика изменения статусов

### Создание анкеты

Когда сиделка впервые создаёт анкету, статус:

draft

### Отправка на модерацию

Когда сиделка отправляет анкету на проверку, статус меняется:

draft → pending

### Одобрение

Когда администратор одобряет анкету, статус меняется:

pending → approved

### Отклонение

Когда администратор отклоняет анкету, статус меняется:

pending → rejected

### Повторная отправка

Если сиделка исправила отклонённую анкету и отправила её снова:

rejected → pending

### Скрытие

Если администратор скрывает уже опубликованную анкету:

approved → hidden

## Что нужно будет реализовать позже

В следующих версиях можно добавить:

- отзывы клиентов;
- рейтинг сиделок;
- проверку документов;
- загрузку сертификатов;
- встроенный чат;
- историю звонков или контактов;
- избранные анкеты;
- жалобы клиентов;
- блокировку пользователей;
- геолокацию;
- уведомления;
- оплату внутри приложения.

Для MVP эти функции не обязательны.

---

## Главный принцип MVP

Сначала нужно сделать простой рабочий механизм:

сиделка создаёт анкету → администратор проверяет → клиент видит одобренные анкеты.

Всё остальное добавляется позже.
