# Схема базы данных MVP v1

## Цель документа

Описать базовую структуру данных для MVP приложения связи сиделок, клиентов и пансионатов.

Схема основана на трёх ключевых сценариях:

- сиделка создаёт анкету;
- администратор модерирует анкету;
- клиент ищет сиделку и связывается с ней.

На этом этапе цель — зафиксировать понятную структуру данных для будущей реализации в Supabase/PostgreSQL.

---

# Главный принцип Supabase

В Supabase уже есть системная таблица пользователей:

`auth.users`

Она отвечает за техническую авторизацию: email, пароль, OAuth, сессии и идентификатор пользователя.

Для прикладных данных MVP используется отдельная таблица:

`public.profiles`

Она хранит роль пользователя, имя, телефон и данные, которые нужны приложению.

## Почему не используем public.users

Название `users` может путаться с системной таблицей Supabase `auth.users`.

Поэтому в прикладной части проекта используем название:

`profiles`

# profiles

Таблица прикладных профилей пользователей.

Она связана с `auth.users` и хранит данные пользователя внутри приложения.

## Назначение

Таблица нужна, чтобы понимать:

- кто пользователь внутри MVP;
- какая у него роль;
- как с ним связаться;
- какой сценарий приложения ему показывать.

## Поля

- id — уникальный идентификатор профиля;
- role — роль пользователя;
- full_name — ФИО пользователя;
- phone — телефон пользователя;
- city — город пользователя;
- created_at — дата создания профиля;
- updated_at — дата последнего обновления.

## Связь с Supabase Auth

`profiles.id` соответствует `auth.users.id`.

Это значит, что после регистрации пользователя в Supabase Auth для него должна создаваться запись в `profiles`.

## Возможные роли

- caregiver — сиделка;
- client — клиент;
- admin — администратор.

## Правила

- один пользователь Supabase Auth должен иметь один профиль в `profiles`;
- роль пользователя хранится в `profiles.role`;
- обычный пользователь может читать и обновлять только свой профиль;
- администратор может видеть профили в рамках задач модерации.

# caregiver_profiles

Таблица анкет сиделок.

В этой таблице хранится информация, которую клиент видит после одобрения анкеты администратором.

## Назначение

Таблица нужна для сценария:

сиделка создаёт анкету → администратор проверяет → клиент видит approved-анкету.

## Поля

- id — уникальный идентификатор анкеты;
- profile_id — ссылка на профиль сиделки из таблицы profiles;
- full_name — ФИО сиделки;
- city — город;
- district — район работы;
- contact_phone — контактный телефон;
- experience — опыт работы;
- education — образование;
- certificates — сертификаты;
- skills — навыки ухода;
- schedule — желаемый график;
- description — краткое описание о себе;
- desired_payment — желаемая оплата;
- ready_for_live_in — готовность к проживанию;
- ready_for_night_shifts — готовность к ночным сменам;
- dementia_experience — опыт с деменцией;
- bedridden_experience — опыт с лежачими пациентами;
- stroke_experience — опыт после инсульта;
- heart_attack_experience — опыт после инфаркта;
- trauma_experience — опыт после травм и аварий;
- photo_url — ссылка на фото;
- status — текущий статус анкеты;
- rejection_reason — причина отклонения;
- hidden_reason — причина скрытия;
- created_at — дата создания анкеты;
- updated_at — дата последнего обновления;
- submitted_at — дата отправки на модерацию;
- approved_at — дата одобрения;
- rejected_at — дата отклонения;
- hidden_at — дата скрытия.

## Связи

`profiles.id → caregiver_profiles.profile_id`

Один профиль с ролью `caregiver` может иметь одну активную анкету сиделки.

## Минимальные обязательные поля для публикации

Анкета сиделки не может получить статус `approved`, если не заполнены:

- full_name;
- city;
- contact_phone;
- experience;
- schedule;
- skills;
- description.

## Важное правило

Обычный пользователь не должен менять поле `status` напрямую.

Статус анкеты меняет только администратор или специальная серверная функция.

# client_requests

Таблица клиентских заявок.

Эта таблица нужна, если клиент не только смотрит список сиделок, но и оставляет заявку на подбор.

На первом этапе MVP эта таблица может быть простой.

## Поля

- id — уникальный идентификатор заявки;
- profile_id — ссылка на профиль клиента из таблицы profiles;
- city — город;
- district — район;
- care_type — тип ухода;
- description — описание ситуации;
- contact_phone — контактный телефон клиента;
- preferred_schedule — желаемый график;
- desired_payment — желаемая оплата;
- needs_live_in — нужен ли уход с проживанием;
- needs_night_shifts — нужны ли ночные смены;
- dementia_case — есть ли деменция;
- bedridden_case — лежачий ли пациент;
- stroke_case — состояние после инсульта;
- heart_attack_case — состояние после инфаркта;
- trauma_case — состояние после травмы или аварии;
- created_at — дата создания заявки;
- updated_at — дата последнего обновления.

## Связи

`profiles.id → client_requests.profile_id`

Один клиент может создать несколько заявок.

# profile_statuses

Справочник статусов анкет сиделок.

## Назначение

Таблица нужна, чтобы статусы были едиными для документации, SQL, Flutter-моделей, RLS и модерации.

## Поля

- code — код статуса;
- title — название статуса;
- description — описание статуса;
- visible_to_client — виден ли этот статус клиенту;
- visible_to_caregiver — виден ли этот статус сиделке;
- visible_to_admin — виден ли этот статус администратору.

## Статусы

### draft

Черновик.

Анкета создана, но ещё не отправлена на модерацию.

Клиент её не видит.

### pending_review

Ожидает модерации.

Сиделка отправила анкету на проверку администратору.

Клиент её не видит.

### approved

Опубликована.

Администратор проверил и одобрил анкету.

Клиент видит эту анкету в списке сиделок.

### rejected

Отклонена.

Администратор отклонил анкету и указал причину.

Клиент её не видит.

### hidden

Скрыта администратором.

Анкета может быть скрыта после публикации из-за жалобы, ошибки, подозрения на спам или нарушения правил.

Клиент её не видит.

# moderation_logs

Таблица истории модерации.

## Назначение

Таблица нужна, чтобы хранить действия администратора по анкетам сиделок.

Без этой таблицы модерация превращается в ручную правку базы без истории действий.

## Поля

- id — уникальный идентификатор записи;
- caregiver_profile_id — ссылка на анкету сиделки;
- admin_profile_id — ссылка на профиль администратора из таблицы profiles;
- old_status — предыдущий статус анкеты;
- new_status — новый статус анкеты;
- reason — причина решения;
- comment — дополнительный комментарий администратора;
- created_at — дата действия.

## Связи

`caregiver_profiles.id → moderation_logs.caregiver_profile_id`

`profiles.id → moderation_logs.admin_profile_id`

## Примеры действий

Администратор может:

- одобрить анкету;
- отклонить анкету;
- скрыть анкету;
- вернуть анкету на доработку;
- повторно одобрить исправленную анкету.

---

# Правила видимости

## Что видит клиент

Клиент видит только анкеты со статусом:

- approved.

Клиент не видит анкеты со статусами:

- draft;
- pending_review;
- rejected;
- hidden.

## Что видит сиделка

Сиделка видит свою анкету в любом статусе:

- draft;
- pending_review;
- approved;
- rejected;
- hidden.

Также сиделка должна видеть причину отклонения, если анкета получила статус `rejected`.

## Что видит администратор

Администратор видит анкеты со статусами:

- pending_review;
- approved;
- rejected;
- hidden.

Также администратор видит историю модерации.

# Логика изменения статусов

## Создание анкеты

Когда сиделка впервые создаёт анкету, статус:

`draft`

## Отправка на модерацию

Когда сиделка отправляет анкету на проверку, статус меняется:

`draft → pending_review`

## Одобрение

Когда администратор одобряет анкету, статус меняется:

`pending_review → approved`

## Отклонение

Когда администратор отклоняет анкету, статус меняется:

`pending_review → rejected`

## Повторная отправка

Если сиделка исправила отклонённую анкету и отправила её снова:

`rejected → pending_review`

## Скрытие

Если администратор скрывает уже опубликованную анкету:

`approved → hidden`

# SQL constraints

## Минимальные ограничения

На уровне базы данных нужно предусмотреть:

- обязательность `profile_id` для caregiver_profiles;
- обязательность `profile_id` для client_requests;
- обязательность `admin_profile_id` для moderation_logs;
- обязательность `status` для caregiver_profiles;
- допустимые значения статусов;
- связь caregiver_profiles.profile_id с profiles.id;
- связь client_requests.profile_id с profiles.id;
- связь moderation_logs.admin_profile_id с profiles.id;
- связь moderation_logs.caregiver_profile_id с caregiver_profiles.id.

## Обязательные поля анкеты

Для публикации анкеты должны быть заполнены:

- full_name;
- city;
- contact_phone;
- experience;
- schedule;
- skills;
- description.

## Важное уточнение

Часть проверок можно делать на уровне приложения, но критически важные правила лучше дублировать на уровне базы данных или серверной функции.

Особенно это касается:

- допустимых статусов;
- запрета изменения status обычным пользователем;
- обязательных полей перед публикацией.

# Пагинация и лимиты выборки

## Почему это важно

Список сиделок не должен загружаться через неограниченный запрос.

Нельзя проектировать клиентский экран так, будто `select('*')` всегда вернёт все анкеты.

Supabase/PostgREST может ограничивать выдачу строк, и при росте базы клиент может получить неполный список без очевидной ошибки.

## Правило для списка сиделок

Клиентский список сиделок всегда загружается постранично.

Минимальная логика выборки:

- status = approved;
- city = выбранный город;
- limit = 20 или 50;
- order by approved_at или created_at;
- pagination: page 1, page 2, page 3.

## Что нельзя делать

Не использовать неограниченный запрос:

`select('*')`

для получения всех анкет из `caregiver_profiles`.

## Что нужно делать

Использовать ограниченную выборку:

- фильтр по статусу;
- фильтр по городу;
- явный limit;
- стабильную сортировку;
- постраничную загрузку через page/range.
